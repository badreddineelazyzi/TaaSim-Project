# ADR-005: Security — JWT Auth, HTTPS, Kafka ACLs

## Status
Accepted

## Date
2026-06-20

## Context
TaaSim handles sensitive driver locations (GPS), rider requests, and admin forecasting tools.
We must protect endpoints against unauthorized access, encrypt transit data, and restrict stream access, while remaining reproducible on local Docker environments.

## Decisions

### Decision 1: FastAPI Role-Based JWT Authentication
We implement HS256 JWT tokens with a 24-hour expiry using a `RoleChecker` dependency injection.

```python
SECRET_KEY = "taasim-secret-key"
ALGORITHM = "HS256"
allow_admin_only = RoleChecker(["admin"])
```
- **Rider role**: Allowed access to `/api/v1/trips`.
- **Admin role**: Allowed access to `/api/v1/vehicles` and `/api/v1/demand/forecast`. Admin-only endpoints return `403 Forbidden` if a rider token is provided.

#### Trade-offs
- Symmetric HS256 is simpler for local development than asymmetric RS256 but requires sharing the secret key across any service that validates tokens directly.

### Decision 2: Self-Signed SSL/TLS Certificates Baked into Docker Image
We use a self-signed cert (`cert.pem`, `key.pem`) generated via OpenSSL. The certs are copied into the FastAPI Docker image during build:
```dockerfile
COPY cert.pem key.pem /app/
RUN chown spark:spark /app/cert.pem /app/key.pem && chmod 600 /app/cert.pem /app/key.pem
```
Uvicorn is configured to start with `--ssl-keyfile` and `--ssl-certfile`.

#### Why baked-in instead of host-mounted?
- Host mounting on Windows paths (`C:\Users\...`) into Linux containers causes permission mismatches. Uvicorn running as `spark` user cannot read `root`-owned windows mounts. Baking the certs and applying `chmod 600` ensures strict read permissions for the application user.

### Decision 3: Permissive Kafka ACLs with StandardAuthorizer
We configure Kraft Kafka with `StandardAuthorizer` (`KAFKA_AUTHORIZER_CLASS_NAME`) and list explicit ACLs for `raw.`, `processed.`, and consumer groups via the ANONYMOUS principal.
Importantly, we keep `KAFKA_ALLOW_EVERYONE_IF_NO_ACL_FOUND=true` (Permissive mode).

#### Why Permissive?
- In single-node KRaft setups, setting `allow.everyone=false` prevents the controller broker from self-registering metadata during startup (since it uses ANONYMOUS internally). 
- Permissive mode allows broker controller registration to succeed, while still enforcing and displaying explicitly defined ACLs for security audits.

## Consequences

### Positive
- All API communication encrypted via HTTPS.
- Endpoint access secured by role.
- Secure, unprivileged Docker API runner (`spark` user, no root execution).
- Auditable Kafka ACL mappings.

### Negative
- Self-signed cert requires clients to use `-k` (insecure) flag for cURL or bypass SSL checks in local testing.
- Kafka ACLs do not strictly block unauthorized anonymous clients because of permissive KRaft fallback requirement.

### Risks and Mitigations
- **Risk**: Hardcoded secret keys.
  *Mitigation*: Use environment variables (`SECRET_KEY`) populated from Docker secrets in production.

## Implementation Notes
- `api/auth.py`: HS256 token generation and `RoleChecker` decorator.
- `docker-compose.yml`: Kafka StandardAuthorizer configs and API HTTPS healthcheck (`curl -kfsS https://localhost:8000/docs`).
- `api/Dockerfile`: Strict cert ownership setting during Docker build.
