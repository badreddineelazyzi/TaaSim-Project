# ADR-002: Event-Time Processing & Checkpointing Strategy

## Status
Accepted

## Date
2026-06-20

## Context
TaaSim processes live taxi GPS events and trip requests through Flink streaming jobs.
We need a robust event-time processing model that handles out-of-order events, provides exactly-once guarantees, and enables fast recovery from failures.

The platform runs on local Docker Compose with Flink 1.20, Kafka (KRaft), and MinIO for checkpoint storage.

## Decision 1: Watermark Strategy
We use bounded out-of-orderness watermarks with different thresholds per job.

### GPS Normalizer — 3 minutes
```java
WatermarkStrategy.<GpsEvent>forBoundedOutOfOrderness(Duration.ofMinutes(3))
```

### Demand Aggregator & Trip Matcher — 1 minute
```java
WatermarkStrategy.<GpsUpdate>forBoundedOutOfOrderness(Duration.ofMinutes(1))
```

### Why different thresholds?
- GPS producer adds jitter and simulates blackouts (up to 180s delay)
- Downstream jobs see already-normalized events with less variance

### Trade-offs
- Longer watermark = more completeness but higher latency
- 3 minutes chosen to cover GPS blackout simulation (60-180s) plus network jitter

## Decision 2: Checkpointing Configuration
All Flink jobs use identical checkpoint settings:

```java
env.enableCheckpointing(60000, CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30000);
env.getCheckpointConfig().setCheckpointTimeout(120000);
env.getCheckpointConfig().setExternalizedCheckpointCleanup(
    CheckpointConfig.ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION);
```

### Why 60-second interval?
- Balance between recovery time (replay from last checkpoint) and checkpoint overhead
- RocksDB incremental checkpoints keep payload small (~20-50 KB per operator)

### Trade-offs
- Shorter interval = faster recovery but more overhead
- EXACTLY_ONCE required for financial (fare) and trip state correctness

## Decision 3: State Backend — RocksDB Incremental
```yaml
state.backend: rocksdb
state.checkpoints.dir: s3a://checkpoints/flink
```

### Why RocksDB?
- Large keyed state (vehicles per zone, pending trips) exceeds heap
- Incremental checkpoints only upload changed SST files
- Local recovery from taskmanager disks when possible

### Trade-offs
- Requires native RocksDB library in container
- Slightly higher CPU for compaction vs heap backend

## Decision 4: Late Event Handling
- **Deduplication**: 2-second window per taxi_id (GPS Normalizer)
- **Idleness**: 30-second timeout for Trip Matcher partitions with no GPS updates
- **Late events**: Watermark allows events up to threshold; later events drop (logged via late data output tag if needed)

## Decision 5: Checkpoint Storage — MinIO (S3A)
Checkpoints stored at `s3a://checkpoints/flink/{job-id}/chk-{n}/`

### Verified Recovery
- Taskmanager restart triggered recovery from chk-65
- All 3 jobs restored state in ~45 seconds
- Post-recovery data counts increased (vehicle_positions: 4034→4455)

## Consequences
### Positive
- Exactly-once end-to-end (Kafka transactions + Flink checkpoints + Cassandra idempotent writes)
- Fast recovery demonstrated in integration test
- Handles simulated GPS blackouts and network reordering

### Negative
- 3-minute watermark adds latency to demand/trip outputs
- RocksDB state requires disk space on taskmanager
- Checkpoint storage grows over time (cleanup policy needed)

### Risks and Mitigations
- Risk: RocksDB state corruption
  Mitigation: `RETAIN_ON_CANCELLATION` + periodic savepoints
- Risk: Watermark too short → late events dropped
  Mitigation: Monitor late event metrics; adjust threshold if needed
