# ADR-004: ML Serving Architecture (FastAPI + Pre-computed Cache)

## Status
Accepted

## Date
2026-06-20

## Context
TaaSim serves live demand forecasts via FastAPI endpoint `POST /api/v1/demand/forecast`.
The ML model (Spark ML `GBTRegressor` PipelineModel trained on 10 months of Porto data) is stored in MinIO (`s3a://ml-data/models/demand_v1/`).
At runtime, requests specify a `zone_id` and a target `datetime`. 

The SLA requires all API endpoint responses to be **< 500ms**. However, initial runs using on-the-fly Spark DataFrame creation and inference took **3.7s - 12s**, violating the SLA due to Spark session compilation and S3 read latency.

## Decisions

### Decision 1: Feature Pre-Loading & Caching at Startup
We load the entire feature dataset (181K rows) from S3 (`s3a://ml-data/features/`) into a local list of Spark `Row` objects once during FastAPI startup.

#### Why
- Reading Parquet from S3 on every HTTP request is highly latent due to network handshakes and S3A connector overhead.
- In-memory lookups reduce data access times from seconds to microseconds.

### Decision 2: Pre-Computing All Zone Predictions at Startup
During startup, we pre-compute the demand prediction for all 16 Casablanca zones for the current time window, using the loaded Spark ML model, and store them in a fast Python dictionary:
```python
predict_df_cache = {zone_id: predicted_demand}
```

#### Why
- Calling Spark ML `demand_model.transform(df)` triggers Catalyst query planning and JVM task scheduling. Even on a single-row DataFrame with a cached schema, Spark ML inference takes **1.5s - 2.5s**.
- Pre-computing and caching the predictions bypasses Spark entirely during HTTP request handling, reducing endpoint latency to **~54ms**.

### Decision 3: Fallback On-The-Fly Inference with StructType Schema
If a zone is missing from the cache, we fall back to on-the-fly Spark DataFrame creation and inference using an explicit `StructType` schema instead of dynamic type inference:
```python
schema = StructType([StructField(c, DoubleType(), True) for c in FEATURE_COLS])
df = spark.createDataFrame([pred_row], schema=schema)
```

#### Why
- Dynamic schema inference (`spark.createDataFrame([row])`) forces Spark to inspect elements and determine types, adding 1-2s of extra latency.
- Explicit schema declaration avoids this reflection overhead.

## Consequences

### Positive
- **SLA Met**: Response time dropped from **12,000ms** to **54ms** (99% latency reduction).
- Highly resilient: Fallback logic handles dynamic forecasting if cache loading fails.
- Low memory footprint: Caching 16 float values is extremely lightweight.

### Negative
- Startup delay: API startup takes **~90 seconds** because Spark must initialize and pre-compute all zone predictions.
- Temporal lag: Predictions are static for the duration of the startup context. If live features change, the API must be re-warmed.

### Risks and Mitigations
- **Risk**: Stale predictions if live data changes.
  *Mitigation*: Implement a background thread in FastAPI that regenerates the 16 predictions every 30 minutes from S3, matching our model's windowing size.

- `api/main.py`: lines 88-100 (startup caching & pre-computation), lines 161-240 (predict_demand endpoint cache-lookup and fallback).
