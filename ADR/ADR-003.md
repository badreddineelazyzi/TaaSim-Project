# ADR-003: Trip Matching Algorithm & State Management

## Status
Accepted

## Date
2026-06-20

## Context
TaaSim matches rider trip requests to available taxis in real time.
The Trip Matcher (Flink Job 3) consumes two streams:
- `processed.gps` — normalized vehicle positions with zone_id, status
- `raw.trips` — rider trip requests with origin/destination zones

We need a low-latency matching algorithm that:
- Finds nearest available vehicle in origin zone or adjacent zones
- Handles vehicle availability changes in real time
- Times out unmatched requests after 5 seconds
- Provides exactly-once trip state in Cassandra

## Decision 1: Co-Processing with KeyedCoProcessFunction
We use `KeyedCoProcessFunction` with both streams keyed by a global constant key (`"global"`).

```java
DataStream<GpsUpdate> gpsStream = ...
DataStream<TripRequest> tripStream = ...

DataStream<MatchResult> matches = gpsStream
    .keyBy(g -> "global")
    .connect(tripStream.keyBy(t -> "global"))
    .process(new TripMatcherFn(zones));
```

### Why global key?
- All vehicles and trips must be visible to a single operator instance for cross-zone search
- Parallelism = 1 (acceptable for 16 zones, ~100s vehicles)
- Simpler than sharding by zone and coordinating across instances

### Trade-offs
- Single-threaded processing limits throughput
- For 16 zones / ~100 vehicles, latency is <50ms per event
- If scale increases, would shard by zone_id and broadcast trips to relevant partitions

## Decision 2: Zone Search Strategy — Origin + Adjacent
When a trip arrives, search zones in order:
1. `origin_zone` (exact match)
2. Adjacent zones from `zone_mapping.csv` adjacency list

```java
Set<Integer> searchedZones = new LinkedHashSet<>();
searchedZones.add(t.originZone);
Zone originZ = zoneMap.get(t.originZone);
if (originZ != null && originZ.adjacent != null) {
    searchedZones.addAll(originZ.adjacent);
}
for (int zid : searchedZones) {
    matchedTaxi = findVehicleInZone(zid);
    if (matchedTaxi != null) { matchedZone = zid; break; }
}
```

### Why adjacency list?
- Realistic: taxis serve nearby zones
- Pre-computed from OSM road network
- Avoids full zone scan (16 zones small, but pattern scales)

## Decision 3: Vehicle Selection — Oldest Available
Within a zone, pick the vehicle with oldest `lastSeen` timestamp (longest idle).

```java
String oldest = null;
long oldestTs = Long.MAX_VALUE;
for (Map.Entry<String, VehicleInfo> e : zv.entrySet()) {
    if (e.getValue().lastSeen < oldestTs) {
        oldestTs = e.getValue().lastSeen;
        oldest = e.getKey();
    }
}
```

### Why oldest idle?
- Fair dispatch (FIFO by availability)
- Reduces driver wait variance
- Simple, no scoring model needed

## Decision 4: Timeout — 5-Second Event-Time Timer
Unmatched trips register an event-time timer for 5 seconds:

```java
pendingTrips.put(t.tripId, t);
ctx.timerService().registerEventTimeTimer(t.timestamp + 5000);
```

On timer fire, trip marked unmatched and removed from pending state.

### Why 5 seconds?
- Meets SLA: match latency P95 < 5s
- Short enough for rider experience, long enough for vehicle to appear
- Event-time timer aligns with watermark (1-min out-of-orderness)

## Decision 5: State Management — RocksDB MapState
Two keyed state structures backed by RocksDB:

```java
vehiclesByZone = MapState<Integer, Map<String, VehicleInfo>>  // zone_id → {taxi_id → VehicleInfo}
pendingTrips = MapState<String, TripRequest>                   // trip_id → TripRequest
```

### Why MapState?
- Efficient point lookups/updates per zone/trip
- RocksDB handles large state off-heap
- Incremental checkpoints to MinIO (60s interval)

### Trade-offs
- MapState serialization overhead (POJO → bytes)
- Keyed by global key → all state in single task, no sharding

## Decision 6: Output — Dual Sink
- **Matched trips** → Cassandra `trips` table + Kafka `processed.matches`
- **Unmatched/expired trips** → Kafka `processed.unmatched`
- All `MatchResult` events include `matched` boolean, `eta_seconds`, `matched_zone`

## Consequences
### Positive
- Sub-5s match latency achieved (measured ~2s P95)
- Exactly-once via Flink checkpoints + Kafka transactions + Cassandra idempotent writes
- Adjacent-zone fallback increases match rate in sparse zones
- Timer-based timeout handles no-vehicle scenarios gracefully

### Negative
- Single-task parallelism limits horizontal scaling
- 5s timeout may drop trips during temporary vehicle gaps
- MapState per-zone iteration on every GPS update (O(zones × vehicles))

### Risks and Mitigations
- Risk: State grows unbounded if trips never time out
  Mitigation: 5s timer cleanup + periodic state TTL (not yet implemented)
- Risk: Single task bottleneck at scale
  Mitigation: Shard by zone_id, broadcast trips to relevant shards
