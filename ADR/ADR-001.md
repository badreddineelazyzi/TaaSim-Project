# ADR-001: TaaSim Platform Architecture

## Status
Accepted

## Date
2026-04-15

## Context
TaaSim is a taxi simulation platform for Casablanca.
We need a system that can:
- process live taxi and trip events,
- keep raw history for replay,
- serve fast query results for dashboards and APIs,
- support offline analytics and ML training.

The team works on local machines with Docker Compose.
So we need tools that are easy to run locally and still match real cloud patterns.

## Decision 1: Kappa Architecture (not Lambda)
We choose Kappa architecture.

### Why
- We use one main data path for events: Kafka -> Flink -> storage.
- This is easier than Lambda because Lambda has two paths (batch + stream).
- One path means less duplicate code and less sync problems.

### Trade-offs
- If we need very complex old-data recompute, Lambda can help.
- But for this project, replay from Kafka archive in MinIO is enough.

## Decision 2: Storage Layer Choices
We use MinIO as data lake and Cassandra as serving database.

### Why MinIO
- S3-compatible object storage.
- Runs locally with Docker.
- Good for raw files, curated files, and ML artifacts.
- Easy to connect from Spark, Flink, and Kafka Connect.

### Why Cassandra
- Very good for high write throughput.
- Good for query-by-partition-key patterns.
- Low-latency reads for zone dashboards and API calls.

### Why not PostgreSQL / MongoDB
- PostgreSQL is strong, but this workload is write-heavy and partition-key driven.
- MongoDB is flexible, but our query model is clearly time + zone based.
- Cassandra fits this pattern better for this phase of the project.

## Decision 3: Cassandra Partition Key Design
We design tables by query pattern, not by normalization.

### vehicle_positions
Partition key: (city, zone_id)

Reason:
- Main query is: available vehicles in one zone.
- This gives fast zone-based lookup.

### trips
Partition key: (city, date_bucket)

Reason:
- Main query is trip history by day.
- date_bucket avoids very large partitions over time.

### demand_zones
Partition key: (city, zone_id)

Reason:
- Main query is live demand heatmap by zone.
- Zone-centric partitioning matches dashboard usage.

## Decision 4: Separation of Concerns (Flink vs Spark)
We split roles between Flink and Spark.

### Flink role
- Real-time stream processing.
- Windowed demand, trip matching, and low-latency updates.

### Spark role
- Offline ETL and feature generation.
- Batch analytics and ML model training.

### Why not only Spark Structured Streaming
- It can do streaming, but Flink is stronger for continuous event-time processing in this design.
- Keeping Spark for batch/ML keeps the system clear and simple.

## Decision 5: MinIO Bucket Structure
We keep clear storage zones.

- raw/: original data and Kafka archive.
- curated/: cleaned and transformed datasets.
- ml-data/: feature sets and model artifacts.

Kafka archive path:
- raw/kafka-archive/raw.gps/
- raw/kafka-archive/raw.trips/

### Why
- Easy data lifecycle management.
- Easy replay for tests and backfill.
- Better governance between source data and processed data.

## Consequences
### Positive
- Simpler architecture with one live event pipeline.
- Good local reproducibility with Docker.
- Fast serving queries from Cassandra by zone/time patterns.
- Replay capability from MinIO archive.
- Clear split between streaming and batch responsibilities.

### Negative
- Team must understand two engines (Flink and Spark).
- Cassandra requires strict query-first schema design.
- Kappa can make large historical recompute slower than a dedicated batch path.

### Risks and Mitigations
- Risk: wrong partition key can hurt performance.
  Mitigation: keep query-driven schema reviews before schema changes.
- Risk: connector/config drift across local machines.
  Mitigation: keep scripts and documented defaults in repository.

## Implementation Notes (Current Project)
- Docker Compose stack includes Kafka, Kafka Connect, MinIO, Cassandra, Flink, Spark, Grafana.
- Kafka Connect S3 Sink mirrors raw topics to MinIO archive.
- Cassandra schema and sample inserts are in scripts:
  - scripts/init_cassandra.cql
  - scripts/insert_samples_cassandra.cql
  - scripts/querys_cassandra.cql
